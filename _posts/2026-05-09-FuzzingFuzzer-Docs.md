## 개요

퍼징(Fuzzing)은 프로그램에 무작위 또는 변형된 입력을 자동으로 대량 주입해 비정상 동작과 취약점을 찾아내는 동적 테스트 기법이다.  
1988년 위스콘신 대학의 Barton Miller 교수가 유닉스 유틸리티에 무작위 입력을 넣어 다수가 크래시되는 현상을 보고하면서 시작된 개념으로, 현재는 가장 효과적인 자동화 버그 발견 기법 중 하나로 자리잡았다.  
Google의 OSS-Fuzz, Chrome, Firefox, 리눅스 커널 등 대부분의 대규모 프로젝트가 보안 검증의 핵심 수단으로 퍼징을 사용한다.  

이 글에서는 퍼징의 기본 동작 원리와 분류, 그리고 대표적인 퍼저 다섯 종류(AFL, AFL++, libFuzzer, honggfuzz, Jackalope)의 차이점을 정리한다.  

## 퍼징의 동작 원리

대부분의 현대 퍼저는 아래 사이클을 반복한다.  

1. 시드(seed) 입력 코퍼스에서 하나를 선택한다.  
2. 입력을 변형(mutation)하거나 새로 생성(generation)한다.  
3. 변형된 입력을 대상 프로그램에 주입해 실행한다.  
4. 실행을 모니터링해서 크래시, 행(hang), 메모리 오류, 새로운 코드 경로 발견 여부를 확인한다.  
5. 가치 있는 입력(새 코드 경로를 만든 입력)은 코퍼스에 추가하고, 크래시는 별도로 저장한다.  
6. 처음으로 돌아가 반복한다.  

이 과정이 1초에 수천 회씩 자동으로 돌아가기 때문에 사람이 발견하기 어려운 엣지 케이스 버그를 찾을 수 있다.  
효율의 핵심은 "어떤 입력이 가치 있는가" 를 어떻게 판단하느냐이며, 이 부분이 곧 퍼저의 종류와 성능을 가른다.  

## 퍼징의 분류

### 입력 생성 방식에 따라

Mutation-based 퍼징은 정상적인 입력 파일(시드)을 비트 플립, 바이트 치환, 청크 삽입/삭제 등으로 변형해 새 입력을 만든다.  
별도의 문법 지식이 필요 없어 적용하기 쉽고, 거의 모든 형식에 두루 쓸 수 있다.  
대신 구조가 매우 복잡한 입력(예: 컴파일러의 소스 코드 입력)에서는 변형된 입력 대부분이 초기 파싱 단계에서 거부되어 깊이 들어가지 못하는 한계가 있다.  

Generation-based 퍼징은 입력의 문법이나 스키마(XML, JSON, 프로토콜 명세 등)에 기반해 처음부터 입력을 생성한다.  
복잡한 구조 입력에 효과적이지만 문법 명세 작성 비용이 추가로 든다.  
syzkaller가 시스템 콜 호출 시퀀스를 명세 기반으로 생성하는 것이 대표적인 예다.  

### 정보 활용 정도에 따라

Black-box 퍼저는 프로그램을 블랙박스로 보고 입출력만으로 동작한다.  
가장 단순하지만 어떤 입력이 어떤 경로로 들어가는지 알 수 없어 효율이 낮다.  

White-box 퍼저는 심볼릭 실행 등을 활용해 모든 경로 조건을 추론하고, 새로운 경로로 가는 입력을 만들어낸다.  
이론적으로 가장 강력하지만 SMT 솔버 비용 때문에 대규모 프로그램에 적용하기 어렵다.  

Grey-box 퍼저는 컴파일 시 계측(instrumentation)이나 동적 계측을 통해 코드 커버리지 정보를 얻고, 새로운 분기를 만든 입력을 다음 시드로 활용한다.  
효율과 비용의 균형이 좋아 현재 대부분의 실용 퍼저가 이 방식을 사용한다.  

### 실행 모델에 따라

Out-of-process(fork) 모델은 매 입력마다 자식 프로세스를 만들어 실행하는 방식이다. 안정적이지만 프로세스 생성 오버헤드가 있다.  
In-process 모델은 같은 프로세스 내에서 타겟 함수를 반복 호출한다. 매우 빠르지만 한 번 메모리가 오염되면 전체가 영향을 받는 단점이 있다.  
Persistent 모델은 둘의 절충안으로, 한 프로세스에서 여러 입력을 처리하다가 일정 횟수마다 재시작한다.  

## 주요 Fuzzer 비교

### AFL (American Fuzzy Lop)

2013년 Michał Zalewski가 공개한 커버리지 기반 퍼저로, 현대 퍼저의 사실상 표준이 되었다.  
컴파일 시 분기 커버리지 추적 코드를 삽입하고 fork-server 모델로 빠르게 입력을 던지는 방식이다.  
원본 프로젝트(google/AFL)는 현재 더 이상 활발히 개발되지 않고, 후속 프로젝트인 AFL++ 가 사실상 그 자리를 이어받았다.  

### AFL++

AFLplusplus 커뮤니티가 유지하는 AFL의 활발한 포크로, 현재 가장 권장되는 범용 퍼저다.  
AFL의 핵심 구조를 유지하면서 다음과 같은 개선이 추가되었다.  

- 더 강력하고 다양한 mutator(MOpt, RedQueen, custom mutator API 등)  
- LLVM/GCC plugin 기반 계측, QEMU/Unicorn/FRIDA 모드 등 폭넓은 백엔드  
- persistent mode, shared-memory 입력 전달 등 속도 개선  
- AddressSanitizer, MemorySanitizer, UBSan 등 sanitizer 통합  
- LAF-INTEL, CmpLog 등 분기 통과 보조 기법  

연구 결과와 커뮤니티 활동이 활발해서 최신 기법이 가장 빠르게 반영된다는 점이 큰 장점이다.  

### libFuzzer

LLVM 프로젝트의 인프로세스 퍼저다.  
타겟 함수 `LLVMFuzzerTestOneInput(const uint8_t *data, size_t size)` 을 작성해 직접 라이브러리에 연결하는 형태로 동작한다.  
별도 프로세스를 띄우지 않아 속도가 매우 빠르고, 라이브러리 단위 퍼징(API 함수 하나, 파서 하나)에 적합하다.  
Clang의 `-fsanitize=fuzzer` 옵션으로 손쉽게 빌드할 수 있고, SanitizerCoverage와 자연스럽게 결합한다.  
OSS-Fuzz 의 기본 엔진 중 하나로 가장 많이 쓰인다.  

### honggfuzz

Google이 만든 멀티스레드 퍼저로, 한 프로세스에서 여러 스레드가 동시에 입력을 던질 수 있다.  
가장 큰 특징은 Intel PT, BTS 같은 하드웨어 기반 커버리지 피드백을 사용할 수 있다는 점이다.  
소프트웨어 계측과 하드웨어 계측 모두 지원해서 소스 컴파일이 가능한 대상부터 소스가 없는 클로즈드 바이너리까지 폭넓게 다룬다.  
역시 OSS-Fuzz 에서 표준 엔진 중 하나로 채택되어 있다.  

### Jackalope

Google Project Zero가 공개한 macOS/Windows 용 클로즈드 소스 바이너리 퍼저다.  
가장 큰 특징은 동적 바이너리 계측(Dynamic Binary Instrumentation, DBI)을 사용한다는 점인데, 소스 코드 없이 실행 중인 바이너리에 계측 코드를 삽입해 커버리지를 얻는다.  
TinyInst 라는 자체 DBI 라이브러리를 사용하고 in-process / out-of-process 모드를 모두 지원한다.  
모듈러 구조라 사용자가 자신만의 mutator, instrumentation, sample delivery 모듈을 끼워넣을 수 있다.  
소스가 없는 상용 소프트웨어(Office, Adobe Reader, Safari 등)의 보안 연구에 자주 활용된다.  

### 비교 요약

| 퍼저 | 주 대상 | 실행 모델 | 커버리지 획득 | 강점 |
|---|---|---|---|---|
| AFL | 소스 컴파일 가능 | fork | 컴파일 계측 | 안정성, 풍부한 자료 |
| AFL++ | 다양 (소스/QEMU/Unicorn/FRIDA) | fork·persistent | 컴파일·QEMU·FRIDA | 범용성, 활발한 개발 |
| libFuzzer | 라이브러리/API | in-process | SanitizerCoverage | 속도, 단위 퍼징 |
| honggfuzz | 다양 | fork·persistent | 소프트웨어·HW(Intel PT/BTS) | 멀티스레드, HW 계측 |
| Jackalope | 클로즈드 바이너리 | DBI 기반 | TinyInst DBI | macOS/Win, 모듈러 |

## 활용 사례

Google의 OSS-Fuzz 는 오픈소스 프로젝트에 libFuzzer, AFL++, honggfuzz 를 함께 운용해 수십만 CPU 시간 단위로 퍼징을 돌린다. 시작 이래 수만 개의 버그를 발견해 보고했다.  
리눅스 커널은 syzkaller 라는 시스템 콜 전용 퍼저로 검증되고, 브라우저 엔진과 미디어 코덱 같은 입력 처리량이 많은 코드는 매일 수십 대 클러스터에서 퍼징된다.  
취약점 연구자 입장에서도 퍼저 한 대를 며칠만 돌려도 알려지지 않은 버그를 발견하는 일이 드물지 않을 만큼 효율이 좋다.  

## 퍼징 적용 시 알아두면 좋은 것

좋은 시드 코퍼스(seed corpus)는 퍼징 효율을 크게 좌우한다. 대상 형식의 다양한 정상 샘플을 적게는 수십 개, 많게는 수천 개 모아 시작해야 한다.  
ASan, MSan, UBSan 같은 sanitizer 와 함께 빌드해 돌리면 단순 크래시뿐만 아니라 use-after-free, uninitialized memory read, integer overflow 같은 메모리 안전성 버그도 잡을 수 있다.  
대상 함수가 너무 무거우면 의미 있는 분기에 도달하기 어렵기 때문에, 하나의 파서나 API 단위로 작게 분리해 fuzz harness 를 작성하는 것이 일반적인 권장 사항이다.  
google/fuzzing 저장소에는 이런 베스트 프랙티스와 튜토리얼이 잘 정리되어 있다.  

## 느낀 점

퍼징을 처음 공부하면서 "무작위 입력을 던지는 단순한 방법" 이라는 인상과 달리, 현대 퍼저들은 커버리지 피드백, 하드웨어 계측, 인프로세스 실행 등 매우 정교한 기법으로 효율을 끌어올리고 있다는 점이 인상적이었다.  
같은 그레이박스 계열이라도 라이브러리(libFuzzer), 소스가 있는 바이너리(AFL++, honggfuzz), 클로즈드 바이너리(Jackalope)처럼 대상에 따라 적합한 도구가 다르다는 것도 새롭게 알게 되었다.  
또한 퍼징은 도구만 돌린다고 끝나는 작업이 아니라, fuzz harness 작성·시드 코퍼스 구성·sanitizer 설정 같은 사전 준비가 결과 품질을 결정한다는 점이 흥미로웠다.
